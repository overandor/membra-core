# Proof-of-Yield Doctrine

## What Proof-of-Yield Is

Proof-of-Yield (PoY) is a **cryptographic attestation** that validators agree on the economic value of a batch of verified work, where value is derived from build artifacts, test results, and external receipts — not from the existence of files alone.

PoY does NOT mean "files are money." It means:

> **Work was performed, artifacts were generated, tests passed, hashes were computed, and validators agreed that the batch has measurable contribution value.**

## The Doctrine

```
Human Intent → LLM Structures → Terminal Executes → Artifacts Generated
     ↓              ↓                ↓                    ↓
  Prompt        Build Plan       Code/Tests         Hashed Output
     ↓              ↓                ↓                    ↓
  ───────────────────────────────────────────────────────────────
                    Proof-of-Yield Consensus
  ───────────────────────────────────────────────────────────────
     ↓              ↓                ↓                    ↓
  Inference     Yield Estimate   Artifact Receipt   Settlement Receipt
     Hash         Hash            (local)            (optional)
```

## What Counts as Yield

| Stage | Description | Verifiable? |
|-------|-------------|-------------|
| **Build Artifact** | File generated by LLM + terminal execution | ✅ Hash of file contents |
| **Test Pass** | pytest/cargo test succeeded | ✅ Exit code + log hash |
| **Contract Compile** | solc/rustc compilation succeeded | ✅ Binary hash |
| **Benchmark Run** | Performance measured with reproducible methodology | ✅ Result JSON hash |
| **Devnet Receipt** | Solana memo transaction confirmed | ✅ Explorer tx signature |
| **External Settlement** | Actual payment received from buyer/protocol | ✅ Invoice / tx hash |

## What Does NOT Count as Yield

| Not Yield | Why |
|-----------|-----|
| LLM text output | Unverified, no execution proof |
| File existence | No work performed on it |
| Strategy idea | No backtest or execution |
| Prompt alone | No artifact generated |
| Internal ledger op | Buffer entry, not consensus |
| Unconfirmed devnet tx | Could fail or be reverted |

## The Three Yields

### 1. Build Yield (Internal)
Artifacts created, tests passed, benchmarks completed. Verified by local hash.

### 2. Consensus Yield (Network)
2/3 of validators agree on inference hash + yield estimate. Verified by consensus proof.

### 3. Financial Yield (External)
Actual payment from buyer, protocol fee, or settlement. Verified by receipt.

**Critical boundary:** Only Financial Yield is real money. Build Yield and Consensus Yield are contribution proofs, not currency.

## Consensus Rules

A batch finalizes when ALL of the following are true:

1. **Minimum validators:** 3+ votes received
2. **Inference agreement:** 2/3 of validators produce the same inference hash
3. **Yield agreement:** 2/3 of validators produce the same yield hash
4. **Artifact evidence:** Batch contains at least one verified build artifact

## Yield Engine

The YieldEngine estimates potential value from file corpus analysis:

- **File type scarcity** (`.sol` > `.rs` > `.py` > `.js`)
- **Code complexity** (functions, imports, lines)
- **Documentation quality** (comment ratio)

These estimates are **scenario projections**, not guaranteed returns. They feed into consensus as one dimension of agreement.

## Honest Claims

| Claim | Truth |
|-------|-------|
| "This file is worth $0.05" | "This file scored 0.05 on our scarcity-complexity metric" |
| "Consensus reached on yield" | "2/3 validators agreed on the same yield hash" |
| "Batch has value" | "Batch contains verified artifacts and validator agreement" |
| "$100/day target" | "If external buyers pay for verified builds, this throughput could theoretically generate $100/day" |
