# MEMBRA Chat-to-Chain Protocol

## The Thesis

MEMBRA is not an SDK. It is a **human value infrastructure** that transforms conversation into verifiable, on-chain proof-of-job.

The chain of transformation:

```
HUMAN CHAT
    ↓
INTENT EXTRACTION (Chat Compiler)
    ↓
STRUCTURED JOB SPEC (Containerized Intent)
    ↓
EXECUTION (Terminal / Docker / Agent)
    ↓
ARTIFACTS (Code, Tests, Docs, Proofs)
    ↓
YIELD SCORING (4 dimensions)
    ↓
VALIDATOR CONSENSUS (2/3 majority)
    ↓
PROOF BUNDLE (Hash-rooted, portable)
    ↓
SOLANA ON-CHAIN RECORD (membra_core program)
    ↓
SETTLEMENT (Payment, Reputation, NFT, Grant, Bounty)
```

## Why This Matters

Current AI systems produce output that evaporates:
- ChatGPT responses have no provenance
- Claude artifacts have no verification
- Local LLM inference has no audit trail
- Code generated by AI has no test + validator + consensus

MEMBRA makes every AI work product **auditable, consensus-backed, and settlement-ready**.

## The Protocol Stack

| Layer | Component | On-Chain | Off-Chain |
|-------|-----------|----------|-----------|
| L1 | Chat | — | Raw prompts (private) |
| L2 | Job Spec | `job_id_hash` | Full spec JSON |
| L3 | Artifact Manifest | `artifact_root` (Merkle) | Source files, builds |
| L4 | Validator Vote | `vote`, `score`, `reason_hash` | Reason text, evidence |
| L5 | Consensus | `yes_votes`, `no_votes`, `result` | — |
| L6 | Yield | `artifact_yield`, `total_score` | Detailed scoring |
| L7 | Settlement | `receipt_hash`, `amount_lamports` | External payment proof |

## State Machine

```
[Created] → [ArtifactsSubmitted] → [VotingOpen] → [ConsensusAccepted]
                                                      ↓
                                               [YieldRecorded]
                                                      ↓
                                               [Settled] → [Closed]

[ConsensusRejected] ← any point if quorum fails
```

## Validator Architecture

### LLMGPT Validator (Python)
- GPT transformer from scratch (PyTorch)
- Reads code artifacts, evaluates structure/security/tests/usefulness
- Produces structured vote: `{vote, score, reason, checks}`
- Submits to Solana via `SolanaValidatorBridge`

### LLMGPT Validator (C++)
- Same transformer architecture, zero dependencies
- Pure C++17, single executable
- Checkpoint-compatible with Python version
- Runs on any machine without Python/PyTorch

### Deterministic Fallback
- Security scan (eval, os.system, secrets)
- Test result verification
- Policy gate check
- No LLM required

## On-Chain Representation

Every stage produces a Solana account:

| Account | PDA Seed | Stores |
|---------|----------|--------|
| `JobAccount` | `["job", creator, job_id_hash]` | Creator, hashes, status |
| `ArtifactManifestAccount` | `["artifact_manifest", job]` | Merkle root, manifest hash |
| `VoteAccount` | `["vote", job, validator]` | Vote, score, reason_hash |
| `ConsensusAccount` | `["consensus", job]` | Quorum result |
| `YieldAccount` | `["yield", job]` | 4 yield scores |
| `SettlementAccount` | `["settlement", job]` | Receipt hash, amount, type |

## Consensus Formula

```
total_votes = yes_votes + no_votes
ratio_bps = (yes_votes * 10000) / total_votes
accepted = ratio_bps >= threshold_bps  // default 6667 (66.67%)
```

All integer arithmetic. No floating point.

## Security Doctrine

> **"LLMs explain and classify. Solana programs enforce state. RPC verifies transactions. Validators vote. Consensus finalizes. Receipts prove settlement."**

- LLMs do not hold funds
- Solana programs do not hallucinate
- Validators do not have root access
- Consensus is not a payment guarantee
- Settlement requires external confirmation
